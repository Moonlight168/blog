面向对和面向过程的区别

回答：

面向过程编程（POP）：面向过程把解决问题的过程拆成一个个方法，通过一个个方法的执行解决问题。

面向对象编程（OOP）：面向对象会先抽象出对象，然后用对象执行方法的方式解决问题。

OOP 开发的程序一般具有下面这些优点：

  • 易维护：由于良好的结构和封装性，OOP 程序通常更容易维护。
  • 易复用：通过继承和多态，OOP 设计使得代码更具复用性，方便扩展功能。
  • 易扩展：模块化设计使得系统扩展变得更加容易和灵活。

Java 中的类加载机制？

回答： 在 Java 中，类加载机制是指在程序运行时，JVM 会根据类的字节码（.class 文件）来加载类，并创建类的实例。

Java 中的八大基本类型？

回答： Java 的八种基本类型包括：byte、short、int、long、float、double、char 和 boolean。

Java 中如何判断一个对象是基本类型还是引用类型？

回答： 在 Java 中，基本类型是值类型，它们直接存储在栈中，而引用类型是引用类型，它们存储在堆中，指向对象。

自动装箱与拆箱了解吗？原理是什么？为什么要这样做？

回答：

装箱：将基本类型用它们对应的引用类型包装起来；

拆箱：将包装类型转换为基本数据类型

原理： 本质上，自动装箱是编译器的语法糖，在编译阶段， Java 会自动在适当位置调用：

  • Integer.valueOf(int) → 装箱
  • intValue()、doubleValue() 等方法 → 拆箱
Integer i = 10;  //装箱
int n = i;   //拆箱

目的： 简化开发，在 JDK 1.5 之前，如果要把 int 放入 List<Integer>，需要手动装箱：

List<Integer> list = new ArrayList<>();
list.add(Integer.valueOf(1));  // 手动装箱
int x = list.get(0).intValue(); // 手动拆箱

什么是字节码?采用字节码的好处是什么?

回答： 在 Java 中，JVM 可以理解的代码就叫做字节码（即扩展名为 .class 的文件），它不面向任何特定的处理器，只面向虚拟机。 Java 语言通过字节码的方式，在一定程度上解决了传统解释型语言执行效率低的问题， 同时又保留了解释型语言可移植的特点。所以，由于字节码并不针对一种特定的机器， 因此，Java 程序无须重新编译便可在多种不同操作系统的计算机上运行。

重写就是当子类继承自父类的相同方法，输入数据一样，但要做出有别于父类的响应时，你就要覆盖父类方法

接口和抽象类的区别，抽象类的作用？

接口和抽象类的区别

  1. 定义不同
    • 接口（Interface）：是一种规范，定义了一组没有具体实现的方法，类通过实现接口来承诺实现这些方法。接口更注重行为的约束。
    • 抽象类（Abstract Class）：是一种不完全实现的类，可以包含抽象方法（没有方法体）和普通方法（有具体实现）。抽象类更注重代码的复用与继承结构。 。
  2. 继承和实现关系： 一个类可以实现多个接口（多继承），但只能继承一个抽象类。

抽象类主要用于：

代码复用：抽象类可以定义公共的成员变量和方法，子类继承后直接复用，减少重复代码。

强制规范：抽象类中的抽象方法定义了一种“必须实现”的行为，子类继承后必须实现这些方法，起到类似接口的约束作用。

模板设计：可以定义一套调用流程的骨架方法（模板方法模式），具体的细节由子类实现。

static 修饰的成员属于类，而非类的某个实例

static 关键字通过将成员与类绑定，提供了全局共享和便捷访问的能力，但需谨慎使用以避免线程安全

浅拷贝复制对象的引用，深拷贝复制对象的内容，即创建一个新对象，然后复制对象中的内容。

什么是序列化和反序列化，序列化的条件是什么？

序列化： 将对象的状态转换为字节流的过程。

反序列化： 将字节流恢复为对象的过程。

条件： 类必须实现 Serializable 接口或者Jackson库序列化

不进行序列化后果：

  • 无法将对象保存到文件或数据库中
  • 无法在网络传输中传递对象，如 RPC（远程过程调用）中，客户端和服务器之间需要通过网络传输对象。
  • 在执行类似于对象持久化、远程通信、对象缓存等操作时，未序列化的对象将无法被处理，从而抛出运行时异常

List, Set, Queue, Map 有什么区别？

回答：

  • List(对付顺序的好帮手): 存储的元素是有序的、可重复的。
  • Set(注重独一无二的性质): 存储的元素不可重复的。
  • Queue(实现排队功能的叫号机): 按特定的排队规则来确定先后顺序，存储的元素是有序的、可重复的。
  • Map(用 key 来搜索的专家): 使用键值对（key-value）存储，类似于数学上的函数 y=f(x)，"x" 代表 key，"y" 代表 value，key 是无序的、不可重复的，value 是无序的、可重复的，每个键最多映射到一个值。集合框

ArrayList 和 数组 有什么区别？

ArrayList基于动态数组实现，相比静态数组Array更灵活：

  1. 容量可变：ArrayList可自动扩容缩容，Array长度固定；
  2. 类型安全：ArrayList支持泛型，Array不支持；
  3. 存储类型：ArrayList仅存对象（需包装基本类型），Array可存基本类型和对象；
  4. 操作便捷：ArrayList提供add、remove等丰富API，Array仅支持下标访问；
  5. 创建差异：ArrayList无需预设大小，Array创建时必须指定长度 。

为什么要使用集合，如何选择？

一、使用集合的原因

  1. 动态扩容：不用提前固定大小，能根据添加元素数量自动扩充容量。
  2. 功能丰富：自带很多方法，像添加、删除、查找等，用起来方便。
  3. 类型安全：通过泛型，规定集合里能放的数据类型，避免出错。
  4. 高性能：Java自带的集合都经过优化，处理数据速度快。

二、选择集合的方法

  1. 按数据特点选
    • 允许重复数据：选List。查找多就用ArrayList，增删多就用LinkedList。
    • 不允许重复数据：选Set。无序就用HashSet，要排序就用TreeSet 。
    • 存键值对数据：选Map。查找快用HashMap，按键排序用TreeMap。
  2. 按线程环境选
    • 单线程：用ArrayList、HashMap等非同步集合，速度快。
    • 多线程：用Vector、ConcurrentHashMap等同步集合，保证数据安全。

说一说ArrayList 的扩容机制

1. 初始容量与默认值

  • 初始容量：创建 ArrayList 时，若不指定大小（如 new ArrayList()），默认初始容量是 0（JDK 8+ 后）。
  • 第一次扩容：当首次添加元素时，容量会被扩展为 10。

2. 扩容触发条件

  • 核心逻辑：当 元素数量超过当前数组长度 时，触发扩容。
  • 示例：若当前容量为 10，添加第 11 个元素时会触发扩容。

3. 扩容步骤（源码解析）

假设原容量为 oldCapacity，扩容分为 3 步：

① 计算新容量

新容量 = oldCapacity + (oldCapacity >> 1)
即 原容量的 1.5 倍（向下取整）。
例如：原容量 10 → 新容量 10 + 5 = 15。

② 检查是否足够

若新容量仍小于所需最小容量（如添加第 16 个元素时，15 不够用），则直接扩容为 所需的最小容量。

③ 检查是否超出最大容量
  • 若新容量超过 Integer.MAX_VALUE - 8，会进一步调整为 Integer.MAX_VALUE。
  • 若超出 Integer.MAX_VALUE，抛出 OutOfMemoryError。

4. 数组复制

扩容后，ArrayList 会通过 Arrays.copyOf() 方法将原数组的元素复制到新数组中，这一步是 性能开销的主要来源。

5. 手动预分配容量

若提前知道数据量（如 1000 个元素），可通过 new ArrayList<>(1000) 指定初始容量，避免多次扩容，提升性能。

回答：
1. 初始容量：默认 0（首次添加时扩容为 10）。
2. 触发条件：元素数量超过数组长度时。

说一说Comparable 和 Comparator 的区别？

回答：

Comparable 和 Comparator 是 Java 中用于对象排序的两个核心接口，它们的区别可以用「自带排序」和「外部策略」来概括：

1. 核心区别

维度ComparableComparator
接口定义java.lang.Comparablejava.util.Comparator
排序逻辑类自身实现 compareTo() 方法，定义「自然排序」规则。在类外部实现 compare() 方法，定义「临时排序」规则。
耦合度排序逻辑与类绑定，侵入性强。排序逻辑与类分离，解耦性好。
使用场景类的默认排序方式（如 String 按字典序）。类已有默认排序，但需要额外排序规则（如逆序、按年龄）。

2. 代码对比

Comparable 示例（自带排序）
// 类实现 Comparable 接口
public class Person implements Comparable<Person> {
    private String name;
    private int age;

    @Override
    public int compareTo(Person other) {
        return this.age - other.age; // 按年龄升序
    }
}

// 使用：直接调用 Collections.sort()
List<Person> people = new ArrayList<>();
Collections.sort(people); // 自动按 Person 的 compareTo() 排序
Comparator 示例（外部策略）
// 类无需实现任何接口
public class Person {
    private String name;
    private int age;
}

// 定义外部 Comparator
Comparator<Person> ageComparator = new Comparator<Person>() {
    @Override
    public int compare(Person p1, Person p2) {
        return p1.getAge() - p2.getAge(); // 按年龄升序
    }
};

// 使用：在排序时传入 Comparator
Collections.sort(people, ageComparator); // 按指定规则排序

3. 关键方法

  • Comparable：
    int compareTo(T o)
    返回值规则：

    • this < o → 返回负数（如 -1）
    • this == o → 返回 0
    • this > o → 返回正数（如 1）

  • Comparator：
    int compare(T o1, T o2)
    返回值规则与 compareTo 一致，但比较的是两个参数。

4. 优缺点

接口优点缺点
Comparable实现后可直接排序，使用简单。排序规则固定，无法灵活切换。
Comparator支持多维度排序，不修改原类。需要额外创建 Comparator 实例。

5. 常见应用场景

  • Comparable：

    • String、Integer 等类默认实现 Comparable。
    • 自定义实体类（如 User、Product）的默认排序。

  • Comparator：

    • 集合临时排序（如 Arrays.sort(arr, comparator)）。
    • GUI 中的表格排序（可切换升序/降序）。
    • 第三方库的排序策略（如 Stream.sorted(comparator)）。

6. 面试高频问题

问：“如何对一个没有实现 Comparable 的类排序？”
答：

  1. 不修改原类：创建外部 Comparator 实现排序规则。
  2. 修改原类：让类实现 Comparable 接口，添加 compareTo() 方法。

总结

  • 用 Comparable：当类需要有一个固定的、默认的排序方式时。
  • 用 Comparator：当需要灵活切换排序规则或对未实现 Comparable 的类排序时。

为什么泛型不能直接使用基本类型？那如何在泛型中使用基本类型？

回答：

泛型在编译时会进行类型擦除，最终所有泛型类型通常都会被替换为 Object，而基本类型不是 Object 的子类，因此无法直接作为类型实参。

使用对应的包装类（如 Integer、Double 等），Java 会自动进行装箱和拆箱操作。

Java中的类型擦除是什么？这样做的目的和后果是什么？

回答：

1. 类型擦除的定义
  • 编译阶段：Java编译器会将所有泛型类型参数（如List<Integer>中的Integer）替换为其上限类型（通常是Object，若有显式边界则为边界类型）。
  • 运行阶段：泛型类型不再包含实际的类型参数信息，例如List<Integer>和List<String>在运行时是相同的类型（List）。
2. 擦除规则示例
① 无边界的泛型类型 → 擦除为Object
class Box<T> {
    private T value; // 编译后：private Object value;
    public T get() { return value; } // 编译后：public Object get()
}
② 有显式边界的泛型类型 → 擦除为边界类型
class Box<T extends Number> { // 边界为Number
    private T value; // 编译后：private Number value;
}
3. 类型擦除的目的

  1. 保持二进制兼容性：
    Java 5引入泛型时，为了不破坏现有代码（如List仍可与List<Object>兼容），选择在编译阶段移除泛型信息。

  2. 简化实现：
    避免为每种泛型类型生成额外的类（如List<Integer>和List<String>无需单独编译），减少字节码体积。

4. 类型擦除的后果
① 运行时无法获取泛型的具体类型参数
List<Integer> intList = new ArrayList<>();
List<String> strList = new ArrayList<>();

System.out.println(intList.getClass() == strList.getClass()); // true
// 运行时两者都是ArrayList，类型参数被擦除
② 不能创建泛型数组
List<Integer>[] array = new List<Integer>[10]; // 编译错误！
// 原因：类型擦除后无法验证数组元素类型（可能存入List<String>）
③ 泛型类型不能是基本类型
List<int> list = new ArrayList<>(); // 错误！
// 类型擦除后需存储Object，但基本类型（如int）不是Object的子类
④ 静态成员不能引用类的泛型参数
class Box<T> {
    private static T value; // 编译错误！静态成员无法使用泛型类型参数
}
5. 桥接方法（Bridge Method）

为了支持泛型的多态性，编译器会生成桥接方法。例如：

class StringBox extends Box<String> {
    @Override
    public String get() { return super.get(); }
}

编译后会生成两个方法：

public String get() { ... } // 用户定义的方法
public Object get() { return get(); } // 编译器生成的桥接方法，覆盖Box<Object>的get()
6. 面试高频回答

Q1：“为什么Java选择类型擦除？”
A1：为了在引入泛型时保持与旧版本代码的二进制兼容性，避免修改现有类库。

Q2：“类型擦除会导致哪些问题？”
A2：

  • 运行时无法区分List<Integer>和List<String>。
  • 不能创建泛型数组（如List<Integer>[]）。
  • 泛型类型参数不能是基本类型（需用包装类）。

Q3：“如何绕过类型擦除限制？”
A3：

  • 通过反射获取泛型信息（如getGenericSuperclass()）。
  • 使用通配符（如List<?>）结合反射操作。

说一说Java中的反射机制？

回答：

反射的定义

反射（Reflection） 是指在运行时动态获取类的信息（如类的结构、方法、字段等）并操作类的对象（如创建实例、调用方法、修改字段值等）的机制。

反射的实现原理

Java 反射的实现基于以下几个核心组件和机制：

1. 类加载机制与 Class 对象

  • 类加载过程：
    当 JVM 首次加载一个类时，会将该类的字节码（.class 文件）读入内存，并创建对应的 Class 对象。每个类在 JVM 中只有一个 Class 对象，它包含了该类的完整结构信息。

  • Class 对象的作用：
    Class 对象是反射的入口点，通过它可以获取类的所有信息（如方法、字段、构造函数等）。例如：

    Class<?> clazz = String.class;
Method[] methods = clazz.getMethods(); // 获取所有公共方法
2. 运行时类型信息（RTTI）

Java 在运行时为每个对象维护一个类型信息，通过 Class 对象和 Object.getClass() 方法可以获取该信息。反射利用这些信息在运行时动态解析和操作类的成员。

3. 反射 API 的核心类

Java 反射 API 主要位于 java.lang.reflect 包下，核心类包括：

  1. Class：表示类或接口，是反射的基础。
  2. Method：表示类的方法，可以动态调用。
  3. Field：表示类的字段，可以动态读写。
  4. Constructor：表示类的构造函数，可以动态创建对象。
  5. Modifier：提供访问修饰符（如 public、private）的信息。

示例：通过反射调用方法

Class<?> clazz = Class.forName("java.util.ArrayList");
Object instance = clazz.getDeclaredConstructor().newInstance(); // 创建 ArrayList 实例

Method addMethod = clazz.getMethod("add", Object.class); // 获取 add 方法
addMethod.invoke(instance, "hello"); // 调用 add 方法
4. 字节码解析与动态代理

  • 字节码解析：
    反射在运行时通过解析类的字节码（如方法签名、字段类型等）来获取类的结构信息。这需要 JVM 提供的底层支持。

  • 动态代理：
    反射可以在运行时生成代理类（如 Java 的 Proxy 类），实现 AOP（面向切面编程）等功能。动态代理通过字节码生成技术在运行时创建新类。

5. 访问权限控制

Java 反射允许通过 setAccessible(true) 打破访问限制，访问和修改私有成员：

Field privateField = MyClass.class.getDeclaredField("privateField");
privateField.setAccessible(true); // 打破 private 访问限制
privateField.set(instance, "new value");

反射的执行流程

  1. 获取 Class 对象：
    通过 Class.forName()、类名.class 或 对象.getClass() 获取目标类的 Class 对象。

  2. 获取成员对象：
    通过 Class 对象获取 Method、Field 或 Constructor 对象。

  3. 执行操作：

    • 调用 Method.invoke() 执行方法。
    • 调用 Field.get()/set() 读写字段值。
    • 调用 Constructor.newInstance() 创建对象。

反射的典型应用场景

  1. 框架开发（如 Spring、Hibernate）：动态创建对象、注入依赖。
  2. 序列化/反序列化（如 JSON 库）：动态读写对象字段。
  3. 测试工具：访问私有方法和字段进行单元测试。
  4. 插件系统：动态加载和使用外部插件。

🔹多线程

⭐Java 中的多线程实现方式有哪些？

回答：

第一种是继承 Thread 类，重写它的 run 方法，然后调用 start() 启动线程。 这种方式简单，但因为 Java 只能单继承，所以用起来有点局限。

第二种是实现 Runnable 接口，写好 run 方法后，把它传给一个 Thread 对象， 再调用 start()，这种方式更灵活，实际开发中更常用。

第三种是实现 Callable 接口，它跟 Runnable 类似，但能有返回值。 一般配合 FutureTask 使用，适合做一些需要结果的任务。

什么是线程安全？如何实现？

回答： 线程安全指在多线程环境下运行结果正确、状态不被破坏。常见方式有：

  • 使用 synchronized、Lock 加锁；
  • 使用线程安全类如 ConcurrentHashMap；
  • 使用原子类（如 AtomicInteger）进行 CAS 操作。

HashMap 和 Hashtable 的区别？

HashMap 是线程不安全的，允许 null 键和值；Hashtable 是线程安全的，不允许 null 键和值。

什么是线程的虚假唤醒？为什么if而用 while？

回答：

在Java的多线程编程中，使用 while 循环而非 if 来检查等待条件是线程安全的关键实践，主要原因是需要防御虚假唤醒（Spurious Wakeup）和条件状态变化。

1. 虚假唤醒（Spurious Wakeup）

  • 定义：线程可能在没有被明确通知（如 notify()/notifyAll()）的情况下从 wait() 中苏醒。
  • 原因：这是Java语言规范允许的行为，通常由底层操作系统或JVM优化导致。
  • 风险：如果使用 if 检查条件，线程被虚假唤醒后会直接跳过条件检查，继续执行后续代码，可能导致状态不一致。

示例（错误写法，使用 if）：

synchronized (lock) {
    if (step != 1) {  // 只检查一次
        lock.wait();
    }
    // 虚假唤醒时，即使step≠1也会执行这里
    printSecond.run();  // 可能在step≠1时执行，导致顺序错乱
}

2. 条件状态可能被其他线程改变

  • 当线程被唤醒时，条件可能已经被其他线程再次改变。
  • 示例场景：
    1. 线程B在 step=1 时被唤醒。
    2. 线程A抢先执行并将 step 重置为0。
    3. 线程B如果只用 if 检查，会直接执行后续代码，导致 step≠1 时仍执行 printSecond。

正确写法（使用 while）：

synchronized (lock) {
    while (step != 1) {  // 每次苏醒后都重新检查条件
        lock.wait();
    }
    // 只有step==1时才会执行
    printSecond.run();
}

多线程编程中为什么避免使用static变量？

核心原因总结

  1. 破坏封装性

    • static变量属于类而非实例，违反面向对象封装原则
    • 状态应与对象实例关联，而非与类关联

  2. 多实例干扰问题

    • 所有实例共享同一static变量
    • 一个实例的操作会影响其他实例的行为
    • 示例：多个OrderService实例会共用同一把static锁

  3. 测试困难

    • 测试用例之间无法隔离
    • 一个测试修改static状态会影响后续测试
    • 难以并行执行测试

  4. 内存泄漏风险

    • 生命周期与JVM相同，可能导致类无法被GC回收
    • 长期持有资源引用

  5. 并发问题加剧

    • 需要额外处理可见性(volatile)和原子性问题
    • 增加代码复杂度

什么是锁？为什么需要锁？

锁是Java中用于控制多个线程对共享资源访问的机制。当多个线程同时访问共享资源时，可能会导致数据不一致或其他并发问题（如竞态条件）。锁通过保证同一时间只有一个线程可以访问共享资源，从而保证线程安全。

Java中有哪些主要的锁机制？

Java中的锁机制主要分为两类：

  • 内置锁（隐式锁）：使用synchronized关键字实现
  • 显式锁：通过java.util.concurrent.locks包中的接口和类实现（如Lock、ReentrantLock）

synchronized关键字的工作原理是什么？

synchronized是Java的内置锁机制，它基于对象头中的Mark Word实现。当一个线程访问被synchronized修饰的代码块或方法时：

  • 它会尝试获取对象的监视器（monitor）
  • 如果监视器被其他线程持有，则当前线程会被阻塞
  • 成功获取监视器的线程在执行完代码后会释放监视器

4. synchronized有哪些使用方式？

synchronized可以用于：

  • 实例方法：锁对象是当前实例（this）
  • 静态方法：锁对象是类的Class对象
  • 代码块：锁对象是括号中指定的对象

示例：

public class SynchronizedExample {
    // 实例方法同步
    public synchronized void instanceMethod() {
        // 同步代码
    }

    // 静态方法同步
    public static synchronized void staticMethod() {
        // 同步代码
    }

    // 代码块同步
    public void syncBlock() {
        synchronized (this) {
            // 同步代码
        }
    }
}

什么是可重入锁？

可重入锁允许同一个线程多次获取同一把锁而不会被阻塞。Java中的synchronized和ReentrantLock都是可重入锁。例如：

public class ReentrantDemo {
    public synchronized void outer() {
        inner(); // 同一线程可以再次获取锁
    }

    public synchronized void inner() {
        // 执行代码
    }
}

ReentrantLock与synchronized的区别是什么？

什么是读写锁（ReadWriteLock）？

读写锁允许多个线程同时读取共享资源，但在写操作时会互斥。Java中的ReentrantReadWriteLock实现了这一机制，它包含：

  • 读锁：允许多个线程同时获取
  • 写锁：同一时间只能被一个线程获取

示例：

import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ReadWriteLockExample {
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private int data = 0;

    public void readData() {
        rwLock.readLock().lock();
        try {
            // 读取数据
        } finally {
            rwLock.readLock().unlock();
        }
    }

    public void writeData() {
        rwLock.writeLock().lock();
        try {
            // 写入数据
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}

什么是公平锁和非公平锁？

  • 公平锁：线程按照请求锁的顺序获取锁，先到先得
  • 非公平锁：线程获取锁的顺序不确定，可能后请求的线程先获得锁

ReentrantLock默认是非公平锁，可以通过构造函数指定为公平锁：

Lock fairLock = new ReentrantLock(true); // 公平锁

什么情况下选择synchronized，什么情况下选择ReentrantLock？

  • 选择synchronized：

    • 代码简单，不需要高级锁特性
    • 优先考虑代码简洁性和可读性

  • 选择ReentrantLock：

    • 需要可中断锁、尝试锁或公平锁
    • 需要使用多个条件变量
    • 对锁的性能有更高要求

什么是CAS（Compare and Swap）？

CAS是一种无锁算法，用于实现原子操作。它包含三个操作数：

  • 内存位置（V）
  • 预期原值（A）
  • 新值（B）

如果内存位置的值与预期原值相匹配，那么处理器会自动将该位置值更新为新值。否则，处理器不做任何操作。Java中的Atomic类（如AtomicInteger）基于CAS实现。

StampedLock是什么？

StampedLock是Java 8引入的一种新的锁机制，它提供了三种模式：

  • 写锁
  • 读锁
  • 乐观读锁

乐观读锁允许在无竞争情况下进行无锁操作，提高了并发性能

锁与原子类的区别是什么？

  • 锁：通过阻塞线程来保证线程安全，适合保护复杂的操作
  • 原子类：基于CAS实现，是非阻塞的，适合简单的原子操作

原子类（如AtomicInteger）通常比锁具有更好的性能，特别是在竞争较少的情况下。

什么是锁的饥饿和公平性问题？

  • 锁饥饿：某些线程长期无法获得锁，导致程序性能下降
  • 公平性：通过公平锁机制确保所有线程都有机会获得锁

公平锁可以减少锁饥饿问题，但会带来一定的性能开销。

乐观锁的典型实现方式有哪些？

回答：

  1. 版本号机制：

    • 每次修改数据时递增版本号。
    • 线程读取数据时记录版本号，更新时检查版本号是否与读取时一致。
    • 示例：数据库表中添加version字段。

  2. CAS（Compare-and-Swap）：

    • 原子操作，三个参数：内存值（V）、预期原值（A）、新值（B）。
    • 如果 V == A，则将 V 更新为 B；否则操作失败。
    • Java中的AtomicInteger类基于CAS实现。

CAS的底层原理是什么？

回答：

  • 硬件支持：CAS依赖CPU的原子指令（如x86的CMPXCHG），确保操作的原子性。
  • Java实现：
    • Unsafe类的compareAndSwap方法是CAS的核心。
    • AtomicInteger等原子类通过Unsafe实现无锁操作。
  • 示例：
    public final boolean compareAndSet(int expect, int update) {
    return unsafe.compareAndSwapInt(this, valueOffset, expect, update);
}

乐观锁有哪些优缺点？

回答：
优点：

  • 无锁开销，无竞争时性能极高。
  • 线程非阻塞，减少上下文切换。
  • 适合读多写少的场景。

缺点：

  • 冲突频繁时，重试会消耗CPU资源。
  • 不适合长事务（长时间持有数据可能增加冲突概率）。
  • 存在ABA问题（需通过AtomicStampedReference解决）。

什么是ABA问题？如何解决？

回答：

  • 问题描述：值从A变为B再变回A，CAS操作误认为值未修改。
  • 解决方案：
    • 版本号：使用AtomicStampedReference，每次修改时增加版本号。
    • 时间戳：记录值的修改时间，确保唯一性。

乐观锁和悲观锁的适用场景分别是什么？

回答：

什么是 Condition 接口？

回答： Condition接口是 Java 并发包（java.util.concurrent.locks）中的一个组件，用于替代传统的Object.wait()、notify()和notifyAll()方法，提供更灵活的线程等待和唤醒机制。

与 Lock 配合使用：Condition实例通过Lock.newCondition()创建，每个Lock可以关联多个Condition对象。

Condition 的核心方法有哪些？

回答：

  • 等待方法：
    • await()：当前线程进入等待状态，直到被唤醒或中断。
    • await(long time, TimeUnit unit)：带超时的等待。
    • awaitUninterruptibly()：不可中断的等待。
    • awaitUntil(Date deadline)：等待到指定时间。
  • 唤醒方法：
    • signal()：唤醒一个等待在该 Condition 上的线程。
    • signalAll()：唤醒所有等待线程。
